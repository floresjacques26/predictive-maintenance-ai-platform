### Q1: Why did you choose machine-level splits instead of the more common random window split? How does window-level splitting cause leakage?

**Answer:**

In predictive maintenance, sensor data is generated by machines over time. A single machine of 2000 timesteps with window_size=50 and stride=10 produces ~195 overlapping windows. Consecutive windows share 49/50 timesteps (98% overlap).

If you split these windows randomly, window 1 (timesteps 0–49) and window 2 (timesteps 10–59) may end up in train and test respectively. The model trains on nearly the identical sensor state of machine X and is tested on the same machine one stride later — it trivially generalises because it has memorised machine X's degradation pattern.

**Machine-level splitting** assigns the entire lifecycle of each machine to exactly one split. The test set contains machines the model has never seen in any operational state. This is the correct generalisation target in production: the deployed model will encounter new machines that weren't in training.

Empirically, window-level splitting inflates F1 by 10–15pp on this dataset. The inflation is highest for models with large temporal receptive fields (LSTM, CNN) that can memorise machine-specific signatures.

The analogy in NLP: splitting dialogue turns randomly vs. holding out entire conversations. Turn-level splitting leaks speaker style; conversation-level splitting tests whether the model generalises to new interlocutors.

### Q2: You use BCEWithLogitsLoss with pos_weight instead of sigmoid + BCELoss. Why? And how do you determine pos_weight?

**Answer:**

**Why BCEWithLogitsLoss:**

`BCELoss(sigmoid(logit), y)` computes `−[y·log(σ(x)) + (1−y)·log(1−σ(x))]`.

At extreme logits (x >> 0 or x << 0), `σ(x)` saturates to 0 or 1, causing `log(σ(x))` to produce −∞ or numerical underflow in float32. PyTorch's `BCEWithLogitsLoss` avoids this by fusing sigmoid and log into a numerically stable form using the log-sum-exp identity:

```
BCEWithLogitsLoss(x, y) = max(x, 0) − x·y + log(1 + exp(−|x|))
```

This never evaluates `exp(large_number)`.

**Why pos_weight:**

With 3% positive rate, a model that always predicts 0 achieves 97% accuracy. The loss is dominated by the 97 negative examples. `pos_weight = n_neg / n_pos ≈ 30` scales the positive term of the loss so that one positive example has equal influence as 30 negative ones.

**How to determine it:**

```python
pos_weight = preprocessor.compute_pos_weight(y_train)
# = (len(y_train) − y_train.sum()) / y_train.sum()
```

It is computed from the training split only — using val or test data to compute it would leak the test distribution into the training procedure.

**Trade-off:** Very high pos_weight causes high recall at the expense of precision (many false alarms). The ablation study in `scripts/ablation_study.py` quantifies this across different ratio values.

### Q3: How does your calibration pipeline work, and when would Platt Scaling fail?

**Answer:**

**What calibration measures:**

A model is calibrated if `P(Y=1 | ŷ=p) = p` for all p. In other words, when the model outputs 0.7, it should be correct 70% of the time. Uncalibrated models can be overconfident (probabilities cluster near 0 and 1) or underconfident (probabilities cluster near 0.5). In predictive maintenance, the maintenance scheduler needs to interpret probability as a true confidence: `P(failure) = 0.2` should mean "this machine will fail one in five times, schedule optional inspection."

**Pipeline:**

1. Train model on training split
2. Collect raw scores on validation split (no label leakage)
3. Fit `PlattScaler` (logistic regression on raw scores vs. labels) on validation
4. Fit `IsotonicCalibrator` on validation
5. Evaluate Brier Score and ECE on the test set for uncalibrated, Platt, and Isotonic

**When Platt Scaling fails:**

- Platt assumes the calibration mapping is sigmoid-shaped (monotone S-curve). If the model's score distribution is multimodal or the mapping is U-shaped, logistic regression will misfit.
- For neural models that already output well-distributed scores, Platt adds little and can slightly overfit the validation set, especially when val is small (<500 samples).

**When Isotonic Regression fails:**

- Isotonic is non-parametric and can perfectly fit any validation curve, but it overfits severely with small validation sets. It requires ≥1000 validation windows for reliable generalisation.
- It makes no guarantee about scores outside the validation range.

**Expected Calibration Error (ECE) formula:**

```
ECE = Σ_b (|B_b| / N) × |mean_confidence(B_b) − accuracy(B_b)|
```

Bins with more samples are weighted more heavily. ECE < 0.05 is generally considered well-calibrated.

### Q4: Your CNN uses dilated convolutions with exponential dilation (1, 2, 4, 8). Explain the receptive field and why this matters for temporal patterns.

**Answer:**

**Receptive field:**

A standard 1D convolution with kernel size k has a receptive field of k timesteps. When you stack layers, receptive fields add (not multiply) unless you use dilation. A dilated convolution with dilation d inserts (d-1) zeros between kernel elements, so each element attends to positions `i, i+d, i+2d, ...`.

For a single ResidualBlock with kernel_size=7 and dilation=d, the receptive field is:
```
RF = d × (kernel_size − 1) + 1 = 7d − 6
```

With 4 blocks at dilations [1, 2, 4, 8], the cumulative receptive field is:
```
RF_total ≈ (7-1) × (1+2+4+8) + 1 = 6 × 15 + 1 = 91
```

With window_size=50, the model can attend to the entire window from any output position.

**Why this matters:**

Bearing degradation progresses over hundreds of timesteps. A failure pattern may require the model to correlate vibration at t=0 with RPM at t=49 (40 timesteps apart). A small receptive field forces the model to learn only local correlations. The exponential dilation doubles the receptive field at each layer, providing multi-scale coverage without quadratic parameter cost.

**Why GlobalAvgPool instead of max pooling or last element:**

- Max pooling discards all temporal structure except the single most extreme activation.
- Taking the last element is fine for LSTM (hidden state is accumulated), but for CNN each output position is locally computed — there is no "memory" flowing from left to right.
- GlobalAvgPool integrates evidence from all time positions equally, which is appropriate when degradation evidence is distributed across the window.

### Q5: How does your cost-sensitive threshold selection work, and how is it different from simply lowering the threshold to increase recall?

**Answer:**

**The problem with "just lower the threshold":**

Naively decreasing the threshold always increases recall (fewer missed failures) but also increases false positives (more unnecessary maintenance). The question is: at what threshold does the *business cost* — not F1 — stop improving?

**Cost-optimal threshold:**

For a given threshold t, the expected cost is:
```
Cost(t) = FP(t) × C_FP + FN(t) × C_FN
```

where FP(t) and FN(t) are the count of false positives and false negatives at that threshold.

We sweep t ∈ [0, 1] and find the argmin.

**Theoretical formula (under calibration assumption):**

Under the Bayes risk framework with a perfectly calibrated model:
```
t* = C_FP × prevalence / (C_FP × prevalence + C_FN × (1 − prevalence))
```

With C_FN=$50k, C_FP=$5k, prevalence=0.03:
```
t* = 5000 × 0.03 / (5000 × 0.03 + 50000 × 0.97) = 150 / (150 + 48500) ≈ 0.003
```

This means: alert if `P(failure) > 0.003`. Extremely aggressive — justified because a single missed failure costs 10× more than a false alarm, and failures are rare (low prevalence denominator).

**Why this differs from just "lowering the threshold":**

The empirical cost curve may have a different shape than the theoretical formula because:
1. The model may not be perfectly calibrated
2. The discrete nature of FP/FN counts creates step-functions in cost
3. At very low thresholds, FP count grows very fast (most examples predicted positive), eventually dominating cost

The `plot_cost_curves()` visualisation in this platform shows the full curve so the practitioner can see the cost landscape, not just the optimal point.

**Cost comparison output:**

```
Model         F1-thresh  Cost@F1  Cost-optimal-thresh  Cost@opt  Saving
LSTM          0.43       $412,500  0.08                 $285,000  +$127,500
```

The saving represents real business value: by choosing the right threshold for the cost structure, the model generates 31% more value than the naive F1-optimal setting.

### Explanation 1: Why Baseline vs. LSTM/CNN — and What Each Tells You

**The purpose of baselines is not just to establish a floor. It is to diagnose what the neural model is actually contributing.**

**What Random Forest + Logistic Regression tell you:**

Both baselines flatten the window to a 1D vector: (window_size × n_features). This destroys temporal order — timestep 1 and timestep 50 are treated as independent features. Yet if these models achieve F1=0.75, they have proven that:

1. The data contains strong marginal signal (individual sensor readings, independent of ordering, are predictive)
2. You don't need a recurrent model to exploit this signal
3. Any neural model that doesn't significantly beat these baselines is not learning temporal dependencies — it is learning the same feature correlations, just with more parameters

**What LSTM adds (when it works):**

The LSTM processes the sequence left-to-right, updating a hidden state at each timestep. Its advantage over baselines is the ability to learn:

- **Order-dependent patterns**: "vibration spikes at t, then RPM drops 5 timesteps later" — a pattern invisible to a flattened window
- **Long-range dependencies**: connecting an early anomaly at t=5 to a degradation trend at t=45
- **Non-linear temporal dynamics**: the degradation signal is not a simple linear trend

If LSTM beats RF by 5–10pp F1 on this data, that margin represents genuine temporal pattern exploitation.

**What CNN adds (when it works):**

The dilated CNN has a larger receptive field than LSTM's sliding hidden state and is much faster to train. It excels at detecting:

- **Periodic patterns**: mechanical vibration resonance shows up as a periodic signal; the CNN detects it via correlation between kernel weights and the signal period
- **Multi-scale patterns**: different dilation levels capture short-range spikes and long-range trends simultaneously

**When baselines win:**

In practice, if the degradation signal is strong (clear monotonic trend in one sensor), flattened features carry most of the information. The neural models' temporal reasoning provides marginal gains. The ablation study reveals this: with the full degradation severity, all models perform similarly; with subtle degradation (low severity), the LSTM's sequential memory provides a larger advantage.

### Explanation 2: How to Prevent Leakage in Industrial Time-Series — Five Failure Modes

Industrial time-series has unique leakage patterns that differ from standard supervised learning. Here are five failure modes and how this platform guards against each.

**1. Window-level random split (temporal leakage)**

*Problem*: Consecutive windows of the same machine share 90%+ of their timesteps. Randomly assigning them to train/test allows the model to memorise machine-specific states.

*Fix*: Machine-level split. Every window from machine X is in the same split.

*Detection*: Compare performance with machine-level vs. window-level splits. Inflation > 5pp indicates window-level leakage.

**2. Scaler fit on full dataset (distributional leakage)**

*Problem*: `StandardScaler.fit(df[sensor_cols])` computes mean and std including test machines. The scaler then encodes information about test machine distributions into the normalised training features.

*Fix*: `scaler.fit(train_df[sensor_cols])` only. Apply `scaler.transform()` to val and test.

*Detection*: Check if `scaler.mean_` changes when you exclude test machines. If yes, you have leakage.

**3. Label computation look-ahead (future leakage)**

*Problem*: `failure_imminent[t] = 1 if failure occurs in [t, t+H]` requires looking at future timesteps to assign the label. This is correct — it is the definition of the target. But computing it over the test window is not leakage, because at inference time the label is simply unknown (you are predicting it).

*Non-issue here*: Labels are computed once from the full series before splitting, and the split is by machine, not by timestep position within a machine. The model never sees the label as a feature.

**4. Preprocessing fitted on validation set (validation leakage)**

*Problem*: If you use val metrics to select which scaler to use AND refit the scaler on train+val before producing test predictions, you have leaked test information (via the design choice of scaler type).

*Fix*: Scaler type is a hyperparameter fixed before seeing val. The saved preprocessor encodes the scaler fitted on training machines only.

**5. Multiple re-uses of test set (researcher degrees of freedom / p-hacking)**

*Problem*: If you iteratively modify the model and re-evaluate on the test set, you are effectively fitting the model to the test set via your design decisions. The final reported metric will be overly optimistic.

*Fix*: The test set is evaluated exactly once per model checkpoint, using a pre-committed evaluation protocol. Bootstrap CI bands quantify uncertainty. Significance tests require correction for multiple comparisons (Bonferroni or Benjamini-Hochberg) when comparing many models.

### Explanation 3: How to Evaluate Reliability Beyond Accuracy — The Calibration Stack

**Why accuracy is insufficient:**

A model that predicts P(failure)=0.99 for every alarm and P(failure)=0.01 for every non-alarm is perfectly accurate (assuming a 0.5 threshold). But if its true calibration is P(failure|ŷ=0.99) = 0.60, then the maintenance planner who sees "99% failure probability" will trust it far more than warranted. This leads to alarm fatigue, unsafe maintenance prioritisation, and eventually loss of trust in the entire system.

**The calibration stack used here:**

**Layer 1: Reliability Diagram**

Plot `mean(ŷ_b)` vs `mean(y_b)` for b=1..B probability bins. A perfectly calibrated model lies on the diagonal. Over-confident models (common in neural nets) cluster at the extremes: bins at [0.8, 1.0] may have only 60% actual failure rate.

**Layer 2: Expected Calibration Error (ECE)**

```
ECE = Σ_b (|B_b| / N) × |conf_b − acc_b|
```

Sample-weighted mean miscalibration. ECE < 0.05 is the typical deployment threshold. Lower is better.

**Layer 3: Brier Score**

```
BS = (1/N) Σ (ŷ_i − y_i)²
```

Strictly proper scoring rule: the minimum is achieved by the true conditional probability. Unlike accuracy, it penalises confident wrong predictions more than uncertain ones. Brier Skill Score = 1 − BS/BS_ref compares against the no-skill baseline (always predicting prevalence).

**Layer 4: Post-hoc Recalibration**

*Platt Scaling*: fit a logistic regression σ(a·ŷ + b) on the validation set. Works best when the raw score is a monotone transform of the true probability (which it is for well-trained models). Computationally trivial.

*Isotonic Regression*: fit a step function that minimises squared error while enforcing monotonicity. More flexible than Platt but requires ≥1000 validation samples to avoid overfitting. Evaluated via cross-validation.

**Layer 5: Decision under calibration**

The cost-sensitive threshold formula `t* = C_FP × p / (C_FP × p + C_FN × (1−p))` depends on the model being calibrated. An uncalibrated model outputting ŷ=0.8 when the true probability is 0.3 will produce a wildly wrong cost-optimal threshold. Calibration is therefore not just a nice-to-have — it is a prerequisite for reliable cost-sensitive decision making.

**What this tells interviewers:**

You understand that a model is not a prediction machine — it is an uncertainty quantification system. Accuracy tells you if you make the right binary call. Calibration tells you if your confidence is trustworthy. In high-stakes industrial applications, the confidence score is often the primary deliverable, not the binary alert.
